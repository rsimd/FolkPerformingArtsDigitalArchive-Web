## 1. 概要

公開サイトに載せるコンテンツのライフサイクル（登録・更新・公開可否）と、権利・同意の記録、公開前の品質担保を高レベルで定義する。CMS の具体製品やワークフロー詳細は本書および親文書の Open Questions で追記する。

## 2. 要件

| 要件 ID | 説明 |
|---------|------|
| F-ADM-01 | コンテンツの登録・更新・公開フラグを管理する仕組みが必要（CMS 自前、Headless CMS、静的生成パイプライン等は未確定） |
| F-ADM-02 | 権利・同意（肖像、録音、楽譜複製）の記録フィールドをデータモデルに含める |
| F-ADM-03 | 公開前レビュー（ファクトチェック）ワークフローは TBD |
| F-ADM-04 | **保存団体意向**とオプション資産（3D 等）の整合。撤回時のマニフェスト・オブジェクト更新 |
| F-ADM-05 | **provenance 欠落チェック**を PR テンプレに含める |

## 4. リスクと緩和（公開前・公開後の運用）

PR ゲートだけでは足りないケースを明示し、実装・運用手順と揃える。

### 4.1 外部 URL 失効（写真・YouTube・譜面ホスト）

常時クロールはコストが高い。**閲覧時**はリンク切れをグレーアウト・注釈表示する。**定期チェック**は月 1 回程度の GitHub Actions 等の軽量 HEAD/GET に限定してもよい（親文書 Open Questions 12）。

### 4.2 悪意ある寄稿・スパム PR

検討候補: GAS の **URL ドメイン・ホワイトリスト**、**1 日あたり PR 数上限**、初回寄稿者は **Issue のみ**で人間確認後にのみ PR 化、など。採用する組み合わせは組織方針で決める（親文書 Open Questions 14）。

### 4.3 権利・自治体・保存会との合意

フォームの許諾チェックと PR 差し戻しだけでは不足しうるケースを Open Questions 15 に残す。実務では**別途合意**（紙・公式メール）を求め、Issue に**参照リンクまたは記録 ID**を必須とする運用を許容する。

### 4.4 団体意向とデータの不一致

`preservation_group_consent` と実際に載せる URL の範囲を PR で突き合わせる（F-ADM-04）。**部分許諾**（演者のみ可・モーション不可等）は `consent_scope` に一致するキーだけを JSON に含め、UI も同範囲に制限する。

### 4.5 `contributor_title_ja` と個人情報

公開 JSON および検索インデックスに載せる文字列から、**連絡先・未公開メモを除外**する（F-SEARCH-03）。肩書きに氏名・所属のフル表記が入りうるため、**サイト上は属性＋肩書の短縮版**、または**匿名 contributor ID** に落とすかをプライバシー方針で決め、フォームで説明する（親文書 Open Questions 13）。

### 4.6 provenance 欠落

F-ADM-05 に加え、GAS が生成する JSON に **必須キーをテンプレ化**し、マージ前に CI またはレビュースクリプトで検証してもよい。

## 5. 実装フェーズで詰める項目（追記用）

- GitHub ブランチ保護、必須レビュー人数
- CMS / パイプラインの選定（静的のみの間は PR ベースで代替可）
- 同意・権利の運用フローと監査ログ
